  <action>
    Run: npm install next-auth@beta @auth/prisma-adapter

    Create lib/auth.ts (note: lib/ at project root, same level as src/):
    ```typescript
    import NextAuth from 'next-auth'
    import { PrismaAdapter } from '@auth/prisma-adapter'
    import Resend from 'next-auth/providers/resend'
    import { prisma } from '@/lib/db'

    export const { handlers, auth, signIn, signOut } = NextAuth({
      adapter: PrismaAdapter(prisma),
      providers: [
        Resend({
          apiKey: process.env.RESEND_API_KEY,
          from: process.env.EMAIL_FROM ?? 'noreply@yourdomain.com',
        }),
      ],
      session: { strategy: 'database' },
      callbacks: {
        session({ session, user }) {
          // With database strategy, user object (not token) is provided
          if (session.user && user) {
            session.user.id = user.id
          }
          return session
        },
      },
      pages: {
        signIn: '/auth/signin',
        error: '/auth/error',
        verifyRequest: '/auth/verify-request',
      },
    })
    ```

    Create src/types/next-auth.d.ts:
    ```typescript
    import type { DefaultSession } from 'next-auth'

    declare module 'next-auth' {
      interface Session {
        user: {
          id: string
        } & DefaultSession['user']
      }
    }
    ```

    Create src/app/api/auth/[...nextauth]/route.ts:
    ```typescript

    export const { GET, POST } = handlers
    ```

<task type="auto">
  <name>Task 2: Add Auth.js Prisma models to schema and run migration</name>
  <files>prisma/schema.prisma</files>
  <action>
    Add the four Auth.js PrismaAdapter-required models to prisma/schema.prisma. These are ADDITIONS to the existing Setting, ProcessedEmail, ActivityLog models — do NOT modify those yet (that is plan 02).

    Add after the existing models:
    ```prisma
    model User {
      id            String    @id @default(cuid())
      name          String?
      email         String    @unique
      emailVerified DateTime?
      image         String?

      // Prepared for Phase 17 (YNAB OAuth tokens — encrypted at rest)
      oauthToken        String?
      oauthRefreshToken String?
      oauthExpiresAt    BigInt?

      // Prepared for Phase 18 (unique forwarding email per user)
      forwardingEmail String? @unique

      accounts      Account[]
      sessions      Session[]
      activityLogs  ActivityLog[]
      settings      Setting[]
      processedEmails ProcessedEmail[]

      createdAt DateTime @default(now())
      updatedAt DateTime @updatedAt
    }

    model Account {
      id                String  @id @default(cuid())
      userId            String
      type              String
      provider          String
      providerAccountId String
      refresh_token     String? @db.Text
      access_token      String? @db.Text
      expires_at        Int?
      token_type        String?
      scope             String?
      id_token          String? @db.Text
      session_state     String?

      user User @relation(fields: [userId], references: [id], onDelete: Cascade)

      @@unique([provider, providerAccountId])
    }

    model Session {
      id           String   @id @default(cuid())
      sessionToken String   @unique
      userId       String
      expires      DateTime

      user User @relation(fields: [userId], references: [id], onDelete: Cascade)
    }

    model VerificationToken {
      identifier String
      token      String   @unique
      expires    DateTime

      @@unique([identifier, token])
    }
    ```

    Note: The User model references ActivityLog[], Setting[], ProcessedEmail[] as future relations. These will cause a Prisma schema validation error because those models don't have userId fields yet. To avoid blocking this plan, add the relation fields to the existing models WITHOUT adding userId itself — just add the back-relation:

    On ActivityLog model, add: `user User? @relation(fields: [userId], references: [id], onDelete: Cascade)`
    On Setting model, add: `user User? @relation(fields: [userId], references: [id], onDelete: Cascade)`
    On ProcessedEmail model, add: `user User? @relation(fields: [userId], references: [id], onDelete: Cascade)`

    BUT these require the userId field to exist on those models. To make the schema valid, add the userId as an OPTIONAL String? field to ActivityLog, Setting, and ProcessedEmail in this plan. Plan 02 will make them NOT NULL and add the backfill migration.

    For Setting: the current PK is `key String @id`. The @@unique([userId, key]) constraint cannot be set until userId is non-null. In this plan, just add `userId String?` and the relation. Plan 02 handles the unique constraint.

    After updating schema.prisma, run:
    ```bash
    npx prisma migrate dev --name add_authjs_models_and_user_id_nullable
    ```

    This creates the migration file and applies it. Existing data is safe because userId is nullable.
