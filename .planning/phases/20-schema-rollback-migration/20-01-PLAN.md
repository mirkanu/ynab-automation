<objective>
Capture a full pre-migration backup and data snapshot before touching anything in the production database.

Purpose: This is the safety net. If the Phase 20-02 or 20-03 migration goes wrong, the pg_dump backup is the only way to restore without data loss. The snapshot captures the exact Setting rows (including SENDERS, CURRENCY_ACCOUNTS, TEST_MODE) and row counts for ActivityLog and ProcessedEmail so the post-migration diff in Plan 20-03 has a ground truth to compare against.

Output: A single snapshot file at `.planning/phases/20-schema-rollback-migration/pre-migration-snapshot.sql` containing: the pg_dump command output path reference, all Setting rows as SQL-escaped values, ActivityLog row count, ProcessedEmail row count, and documented restore instructions.
</objective>

  <action>
Run the following commands via `railway run` (the Railway CLI proxies the private DATABASE_URL). Collect all outputs into the snapshot file.

Step 1 — Take a full database backup:
```
railway run pg_dump "$DATABASE_URL" --format=custom --file=/tmp/pre-migration-backup.dump
```
Note: The dump file is written inside the Railway container and is NOT persistent. We must extract the content. Instead, use `--format=plain` and capture stdout to a local file:
```
railway run pg_dump "$DATABASE_URL" --format=plain --no-owner --no-acl > .planning/phases/20-schema-rollback-migration/pre-migration-backup.sql
```
This produces a plain SQL dump locally (Railway pipes stdout to local).

Step 2 — Snapshot current Setting rows (key + value, ordered by key):
```
railway run psql "$DATABASE_URL" -c "SELECT key, value FROM \"Setting\" ORDER BY key;" --no-align --tuples-only
```
Capture this output into the snapshot file.

Step 3 — Count ActivityLog rows:
```
railway run psql "$DATABASE_URL" -c "SELECT COUNT(*) FROM \"ActivityLog\";"
```

Step 4 — Count ProcessedEmail rows:
```
railway run psql "$DATABASE_URL" -c "SELECT COUNT(*) FROM \"ProcessedEmail\";"
```

Step 5 — Write `.planning/phases/20-schema-rollback-migration/pre-migration-snapshot.sql` with this structure:
```
-- PRE-MIGRATION SNAPSHOT
-- Captured: {timestamp}
-- Purpose: Ground truth for post-migration verification (Phase 20-03 diff)

-- BACKUP FILE
-- Location: .planning/phases/20-schema-rollback-migration/pre-migration-backup.sql
-- Restore command (if migration fails):
--   railway run psql "$DATABASE_URL" < .planning/phases/20-schema-rollback-migration/pre-migration-backup.sql
-- WARNING: Restore will fail if tables already exist. Drop them first or use pg_restore with --clean.
-- Alternative restore: railway run psql "$DATABASE_URL" -c "DROP SCHEMA public CASCADE; CREATE SCHEMA public;" && railway run psql "$DATABASE_URL" < .planning/phases/20-schema-rollback-migration/pre-migration-backup.sql

-- SETTING ROWS (pre-migration)
-- {paste the psql output here, one row per line: key | value}

-- ACTIVITYLOG ROW COUNT: {N}
-- PROCESSEDEMAIL ROW COUNT: {N}

-- VERIFICATION QUERIES (run post-migration to diff):
-- SELECT key, value FROM "Setting" ORDER BY key;
-- SELECT COUNT(*) FROM "ActivityLog";
-- SELECT COUNT(*) FROM "ProcessedEmail";
```

IMPORTANT: Do not proceed past this task until both `pre-migration-backup.sql` AND `pre-migration-snapshot.sql` exist on disk with real data (not placeholder text).
