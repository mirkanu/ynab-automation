Model 1 — EmailForwardingAddress:
```prisma
model EmailForwardingAddress {
  id          String   @id @default(cuid())
  userId      String
  mailboxHash String   @unique
  email       String

  user        User     @relation(fields: [userId], references: [id], onDelete: Cascade)

  createdAt   DateTime @default(now())

  @@unique([userId, mailboxHash])
  @@index([userId])
}
```

Model 2 — ProcessedWebhook:
```prisma
model ProcessedWebhook {
  id         Int      @id @default(autoincrement())
  provider   String   // "postmark"
  providerId String   // MessageID from provider (unique per delivery)
  userId     String
  status     String   @default("success")  // "success", "error", "skipped"

  user       User     @relation(fields: [userId], references: [id], onDelete: Cascade)

  createdAt  DateTime @default(now())

  @@unique([provider, providerId])
  @@index([userId, createdAt])
}
```

Add back-relations to User model:
```prisma
model User {
  // ...existing fields...
  emailForwardingAddresses EmailForwardingAddress[]
  processedWebhooks        ProcessedWebhook[]
}
```

The `@@unique([provider, providerId])` on ProcessedWebhook is CRITICAL — it is the atomic idempotency guard preventing duplicate YNAB transactions.

After generation, open the migration file and append the RLS policy SQL at the end (after all CREATE TABLE and ALTER TABLE statements):

```sql
-- Enable Row Level Security on new tables (matching Phase 16-03 pattern)
ALTER TABLE "EmailForwardingAddress" ENABLE ROW LEVEL SECURITY;
ALTER TABLE "EmailForwardingAddress" FORCE ROW LEVEL SECURITY;
CREATE POLICY user_isolation ON "EmailForwardingAddress"
  USING ("userId" = current_setting('app.user_id', true));

ALTER TABLE "ProcessedWebhook" ENABLE ROW LEVEL SECURITY;
ALTER TABLE "ProcessedWebhook" FORCE ROW LEVEL SECURITY;
CREATE POLICY user_isolation ON "ProcessedWebhook"
  USING ("userId" = current_setting('app.user_id', true));
```

Important: Use FORCE ROW LEVEL SECURITY (matching Phase 16-03 pattern — FORCE ensures isolation even for superuser roles like the Railway DB user). The RLS policy uses `current_setting('app.user_id', true)` which is set by `getPrismaForUser()` in db.ts.

Note: The webhook route (`/api/email/inbound`) does not use a user session — it uses the userId extracted from MailboxHash. For that route, use `getPrismaForUser(userId)` with the extracted userId so RLS is satisfied. The `ProcessedWebhook` table needs a special case for the 'unknown' userId (orphaned webhook for unknown recipients) — use `prisma` (global, no RLS) for that insert only, since there's no userId to scope it to.
