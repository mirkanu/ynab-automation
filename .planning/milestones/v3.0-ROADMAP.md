## Accomplishments

1. **Config-driven routing** — `SENDERS` JSON env var replaces hardcoded sender map; `CURRENCY_ACCOUNTS` and `ADMIN_EMAIL` replace all personal-prefixed vars
2. **Zero personal references** — every name, email, and personal env var removed from source, tests, and comments
3. **Generic README** — rewritten as a complete third-party setup guide with `config.example.json` format reference
4. **Live migration** — Railway deployment migrated to new env vars; old personal vars deleted; repo made public
5. **Setup wizard** — interactive browser UI at the app URL when unconfigured; connects to YNAB API, builds config, auto-applies vars to Railway via GraphQL API
6. **Railway deploy button** — one-click deploy from README; wizard eliminates manual env var setup for new users

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| SENDERS as JSON env var | Works natively on Railway/PaaS; supports any number of senders |
| CURRENCY_ACCOUNTS as JSON env var | Same pattern as SENDERS; optional; clean generalisation |
| loadConfig() at handler entry | Config errors surface on every request with clear message |
| YNAB API called from browser (not server) | CORS supported; token never hits the Next.js server |
| Single-component wizard (no URL routing) | Simpler state management for a one-time setup flow |
| Railway API auto-apply | Uses built-in RAILWAY_PROJECT/ENV/SERVICE_ID — no manual var copying |
